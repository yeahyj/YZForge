# 06. Editor 插件与 Validator

## Editor 插件定位

Editor 插件是 YZForge 的半个核心。它负责创建、生成、校验，减少手工犯错。

## 创建结构

创建 Module：

```text
assets/modules/<ModuleName>/module.json
assets/modules/<ModuleName>/code/entry.generated.ts
assets/modules/<ModuleName>/code/public.ts
assets/modules/<ModuleName>/code/<ModuleName>Module.ts
assets/modules/<ModuleName>/res
```

`code/entry.generated.ts` 是生成入口，业务不手改。手写业务逻辑目录由项目自行组织；模板默认提供 `<ModuleName>Module.ts`、`flow/`、`service/`、`model/` 作为参考。

创建 Library：

```text
assets/libraries/<LibraryName>/library.json
assets/libraries/<LibraryName>/code/entry.generated.ts
assets/libraries/<LibraryName>/code/public.ts
assets/libraries/<LibraryName>/res
```

Library 的 `code/entry.generated.ts` 也是生成入口，负责注册 `LibraryEntry`。

创建 ContentPack：

```text
assets/content-packs/<OwnerModule>/<ContentPackName>/content-pack.json
assets/content-packs/<OwnerModule>/<ContentPackName>/manifest.generated.json
assets/content-packs/<OwnerModule>/<ContentPackName>/res
```

ContentPack 创建时必须选择 owner Module。没有 owner 的内容包不允许存在。

创建 Flow：

```text
code/flow/HomeFlow.ts
```

这是默认模板路径。项目可以通过生成器配置调整手写代码目录。

创建 View：

```text
code/view/PageHome.ts
code/view/refs/PageHome.refs.generated.ts
res/view/PageHome.prefab
```

这是默认模板路径。只要生成器能建立 prefab、View 类和 AutoRefs 文件之间的映射，手写 View 脚本可以放在项目自己的目录结构里。

## 生成器原则

- generated 文件可覆盖。
- 手写文件不覆盖。
- 每次生成前校验目录。
- 生成后触发 TypeScript 检查。
- 生成产物必须稳定排序，减少 git diff 噪音。
- 生成产物必须带 generator version 和 content hash。
- `ModuleRef`、`LibraryRef` 只能生成到 `assets/app/registry`。
- Contract 只能生成到 `assets/app/contracts`。
- `ModuleEntry`、`LibraryEntry` 只能生成到目标 Scope 的真实 Bundle 内，默认写入 `code/entry.generated.ts`。
- Import Maps 和 `tsconfig.paths` 必须同步生成。

generated 文件头：

```ts
// <auto-generated>
// generator: yzforge@0.1.0
// source: assets/modules/Battle/module.json
// hash: 3d2f...
// </auto-generated>
```

Validator 通过 hash 检查 generated 文件是否被手改。

## 生成顺序

```text
1. scan scope descriptors
2. validate descriptor schema
3. generate contracts from public.ts
4. generate refs into app/registry
5. generate entries into scope code/entry.generated.ts
6. generate asset manifests
7. generate config types
8. generate UI refs
9. generate ContentPack manifests
10. generate import-map.json
11. update tsconfig paths
12. run TypeScript check
13. run Architecture Validator
```

如果任何一步失败，后续步骤不继续，避免半生成状态污染项目。

## Import Maps

生成器维护：

```text
import-map.json
tsconfig.json compilerOptions.paths
```

推荐映射：

```json
{
  "imports": {
    "yzforge/": "./extensions/yzforge/runtime/",
    "yzforge/modules/": "./assets/app/registry/modules/",
    "yzforge/libraries/": "./assets/app/registry/libraries/",
    "yzforge/content-packs/": "./assets/app/registry/content-packs/",
    "yzforge-contracts/": "./assets/app/contracts/",
    "yzforge-shared/": "./assets/shared/code/"
  }
}
```

规则：

- 业务代码只使用稳定别名，不写跨 Scope 相对路径。
- Import Maps 是运行时解析规则。
- `tsconfig.paths` 是 TypeScript 检查规则。
- 两者必须由同一个生成器源数据生成。

## Architecture Validator

必须检查：

1. 目录结构是否符合规范。
2. 模块名、Library 名、ContentPack 名是否 PascalCase。
3. 文件和目录是否存在大小写冲突。
4. `module.json`、`library.json`、`content-pack.json` schema 是否正确。
5. Scope 对应 Bundle 命名是否正确。
6. Cocos Bundle meta 是否存在，Bundle 名是否与描述文件一致。
7. 动态 Bundle 是否包含 `code/entry.generated.ts`。
8. `entry.generated.ts` 是否只 import 当前 Scope、`yzforge`、首包 contract/registry 和 shared。
9. `module.json`、`library.json`、`content-pack.json` 中声明的依赖是否存在。
10. 是否有业务模块直接 import 另一个业务模块。
11. 是否有业务模块 import 另一个模块的 `assets.generated.ts`。
12. 是否有业务模块 import 另一个模块的 `config.generated.ts` 或运行时 config。
13. 是否有业务模块直接 import 另一个模块的 `code/entry.generated.ts`。
14. 是否有业务模块直接 import Library 内部代码。
15. Library 是否 import 了 Module 内部代码。
16. Library 之间是否存在循环依赖。
17. `shared` 是否访问 `global` 或任何业务模块。
18. `global` 内部是否 import Module 或 Library 内部代码。
19. `registry` 和 `contracts` 是否 import 目标 Bundle 运行时值。
20. 是否有手写动态资源路径。
21. 是否有业务代码直接调用 `assetManager.loadBundle`。
22. 是否有业务代码手动 `instantiate` 后未登记 owner。
23. Model 是否 import `cc`。
24. Service 是否直接打开 UI。
25. Service 是否长期持有 `Node` 或 `Component` 类型字段。
26. generated 文件是否被手改。
27. Main 场景是否为启动场景。
28. UI 层节点是否齐全。
29. View prefab 是否挂载对应 View 类。
30. Part prefab 是否挂载对应 Part 类。
31. AutoRefs 引用是否能解析到节点和组件。
32. `res/` 是否包含业务脚本源码。
33. Module prefab 挂载的脚本是否来自本 Module、global public、shared 或已声明 Library。
34. Library 是否只通过 `code/entry.generated.ts` 注册公开 API。
35. ContentPack 是否包含业务脚本源码。
36. ContentPack 的 owner Module 是否存在。
37. ContentPack prefab 挂载脚本是否来自 owner Module、shared 或已声明 Library。
38. ContentPack 的 `manifest.generated.json` 是否存在并与 `res/` 一致。
39. Module 是否声明了自己使用的 Library。
40. Module 是否直接访问了非 owner 的 ContentPack。
41. `ModuleRef` 与 `ModuleEntry` 的依赖声明是否一致。
42. `LibraryRef` 与 `LibraryEntry` 的依赖声明是否一致。
43. `code/public.ts` 是否只导出类型契约，不包含运行时值、`cc` 导入或业务实现。
44. 生成的 Contract 是否不引用目标 Bundle 运行时实现。
45. Import Maps 与 `tsconfig.paths` 是否一致。
46. `assets/shared/res` 如果启用 Bundle，优先级是否高于依赖它的业务 Bundle。
47. ViewRef 是否生成了合法 `ViewPolicy`，且 `ViewKind`、`ViewLayer`、prefab 命名没有明显冲突。
48. Page 是否在同一 owner 内满足互斥策略，Popup/Paper 是否声明了可入栈策略。
49. Module 是否只打开本 Scope View 或通过 Global public API 请求 Global UI。
50. 是否有业务 View 长期持有其他 View 的节点、组件或跨模块 UI handle。
51. View 是否绕过 `this.listen`、`addDisposer` 长期注册事件、计时器或 tween。
52. ContentPack 是否提供了 Page、Paper、Popup 这类 UIManager View。
53. System UI preset 是否存在，且不被 Module 直接持有节点。
54. `openForResult` 是否用于 Toast 或没有结果策略的 View。

## 检查方式

Validator 不只做字符串扫描，至少需要四类检查：

```text
descriptor check   读取 json schema
ast check          解析 TypeScript import/export/type usage
prefab check       读取 prefab 序列化脚本引用和节点标记
asset check        扫描 meta、bundle 配置、资源路径和大小写
```

对无法完全静态证明的规则，必须定义可执行边界。

示例：

- “Service 长期持有 Node”检查类字段和构造缓存，不检查方法局部变量。
- “手写资源路径”检查 `resources.load`、`bundle.load`、`assetManager.loadBundle` 和字符串白名单。
- “generated 被手改”检查文件头 hash。
- “Prefab 脚本来源”检查脚本 UUID 对应资产路径。
- “entry 是否入包”检查 `entry.generated.ts` 所在目录、Bundle meta 和构建配置。
- “手动实例化未登记”检查 `instantiate(...)` 调用是否进入 `assets.instantiate`、`ui.open` 或 `trackNode` 白名单。
- “ViewPolicy 是否一致”检查生成的 ViewRef、prefab 命名、挂载脚本和 UI layer 配置。
- “View 事件未登记”检查 `node.on`、`schedule`、`tween` 等调用是否被 `listen` 或 `addDisposer` 包裹。

## 禁止方向

```text
model -> view
model -> service
model -> cc
service -> ui
service -> view node field
view -> other view node field
module A -> module B internal code
module A -> module B view
module A -> module B assets.generated
module A -> module B config.generated
module A -> module B code/entry.generated.ts
module A -> module B content pack
module -> library internal code
library -> module
content pack -> business source code
shared -> global
shared -> module
global internal -> module internal
registry/contract -> target bundle runtime value
dynamic bundle -> another dynamic bundle internal script
```

## CI 建议

第一版就应该提供命令行校验：

```text
yzforge generate --check
yzforge validate
yzforge validate --strict
```

规则：

- `generate --check` 只检查 generated 文件是否最新，不写文件。
- `validate` 阻断明显架构错误。
- `validate --strict` 打开实验性规则和更强检查。
- CI 必须运行 `generate --check` 和 `validate`。

Validator 返回值同时保留面向脚本的 `issues: string[]` 和面向编辑器面板的 `issueDetails`。`issueDetails` 至少包含 `message`、`code`、`severity`，能解析到资源时还包含 `path`、`url`，AST import 检查可额外提供 `line`、`column`、`specifier` 和 `target`。
